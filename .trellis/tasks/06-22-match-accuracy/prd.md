# 匹配准确度修复（子任务一 · 先做）

> 父任务：`06-22-offerpilot-reshape`。改的是分析内核，与"一波流"正交。

## Goal

让匹配诊断的"当前匹配度 / 要补的关键技能"变得**准确、可解释、可复现**——它是产品"触发期"的门面，也是面试最容易被追问的部分。当前它会给出明显失真的结论，必须先修。

## 问题定位（已通过读码核实，来源：docs/问题清单.md P1–P3）

- **P1 · 打分公式失真**：一份过得去的简历只打 17%。根因在打分逻辑——弱证据只记 0.5 分、必备技能占 80% 权重、且"必备得分 = 命中数 ÷ 必备总数"，**必备技能一多，分母就被撑大、分数被稀释**。
- **P2 · 技能库死板且有语义错误**：词库只有约 30 个词；把 Redis、MongoDB 错误地合并成"关系型数据库"（二者根本不是关系型），把 FastAPI、Flask、Django 全揉成"Python"。**关键**：这个"揉平"发生在**抽词之后的归一化**——所以哪怕配了大模型、正确抽出了 Redis/FastAPI，下游照样揉平，**配不配密钥都错**。
- **P3 · 加分技能被去重误删**：Redis 被折叠成"关系型数据库"后，与已进必备的 SQL 撞车，被"加分与必备重复就删"的规则当成重复删掉，导致 Redis 在结果里凭空消失。是 P2 的连带后果。

（P4 规则模式整行分类过粗、P5 规则模式项目经历未用于强证据，列为次要，是否纳入见下方设计岔路。）

## Requirements（要达到的效果，非实现）

- 打分结果**符合直觉**：一份明显匹配岗位的简历不应给出离谱低分；分数随"命中关键技能比例"单调合理变化，不被必备技能数量稀释。
- 技能识别**语义正确**：不同技能（如 Redis 与 SQL、FastAPI 与 Python）不被错误合并；不再出现"已掌握的技能凭空消失"。
- **可解释**：每个匹配/缺口结论能说清依据（命中了哪些证据、来自简历哪一段），便于在报告里展示、也便于面试自圆其说。
- **可复现**：同一份简历 + 同一组岗位，多次跑结果稳定一致（这条直接约束"要不要全交给大模型"的取舍）。
- **离线可跑**：未配密钥的规则模式仍能跑通（保持现有"无密钥也能演示闭环"的特性），但其质量缺陷需明确区分对待。

## 设计决策（已敲定，2026-06-23 → 详见 design.md「五层架构」）

1. **打分公式**：**重做一套**（去必备数量稀释、按证据强度加权）。
2. **技能库**：**大模型抽取 + 受控归一**；归一层升级为"**大模型判断映射 + 钉进一张活表**"——大模型当大脑、表当结晶记忆，套用查表（可复现），仅遇新词才调大模型。
3. **大模型介入评分**：权威分仍由**规则**给（可复现）；大模型**回头复核**分与证据（握对话历史 + 联网最新数据），产出「AI 复核意见」+ 结构化「问题记录」，问题记录**受控**回流去优化②③的规则。
4. **P4/P5**：本轮**一并修**（规则模式整行分类 + 项目经历强证据）。

> 两个默认待 review 时确认：⑤走"受控合入"（非运行时自动改）；展示一个权威分 + 文字复核意见（不另给"AI 参考分"数字）。

## Acceptance Criteria

- [ ] 打分公式修正后，用一组代表性简历×岗位样例验证：明显匹配的不低于合理区间、明显不匹配的偏低，分数不被必备数量稀释（需在设计阶段定下样例与期望区间）。
- [ ] 技能语义错误清零：Redis/MongoDB 不再归为关系型；FastAPI/Flask/Django 不再被无差别揉成 Python；加分技能不再被误删。
- [ ] 结论可解释字段齐全（命中证据、来源段落）并能在现有报告结构里呈现。
- [ ] 可复现性达标：同输入多次运行结果一致。
- [ ] 现有后端测试通过；为修复点新增针对性用例。
- [ ] 改动写入 `docs/修改日志/`。
