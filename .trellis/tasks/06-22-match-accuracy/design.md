# 匹配准确度修复 · 技术设计（design.md）

> 父任务 `06-22-offerpilot-reshape` 的子任务一。与"一波流"正交。
> 本设计已锁定的取舍：① 大模型抽取；② 大模型判断映射 + 钉进活表；③ 重做评分；④⑤ 大模型复核回路；P4/P5 一并修。

## 0. 一句话设计哲学

**大模型负责"看见(抽取)"和"判断(复核)";规则负责"对齐(归一)"和"算分(评分)"。大模型的灵活成果，永远落到一张人能审、能改、可复现的表上。**

权威分数永远由规则给出（可复现、可解释、可测）；大模型不直接打分，只做抽取、归一判断、与回头复核。

---

## 1. 五层架构与落点

```
输入：简历 + 一组岗位
 ①抽取层(大模型, 降级规则)   resume_parser / jd_parser    抽技能, 带证据强度
 ②受控归一层(规则查表 + LLM补缺)  services/skills.py + data/skills.py + 新归一表  对齐到规范名 + 关系
 ③评分层(规则·重做)          gap_analysis.py              命中×证据强度算权威分
 ④复核层(大模型)             agent.py(analyze_match 之后)  复核分与证据, 出意见 + 问题记录
 ⑤规则优化(受控)             新: 问题记录 → 评审 → 改②③    让表与评分越来越准
```

各层与现有编排的关系：①②③ 全部发生在 `pipeline.analyze_match_streaming`（确定性内核，可复现）；④⑤ 发生在它**之后**、回到 `agent.py` 对话循环里（此时大模型握着对话历史 + 联网数据）。

## 2. 逐层技术设计

### ① 抽取层（大模型主力 / 规则降级）

**现状**：`resume_parser._parse_with_llm` / `jd_parser._parse_with_llm` 让大模型吐技能词（原文写法），但**只吐词、不吐证据强度**；强弱后续靠 `gap_analysis._project_skill_keys` 用 `match_skills` 重新猜（脆弱，P5 根源之一）。

**改造**：让抽取直接产出**带证据的技能项**，强弱在抽取时就定，不再下游重猜：
- 简历每个技能：`{raw, evidence_text, strength}`，`strength ∈ {project(项目实证), work(工作经历), listed(仅罗列)}`。
- 岗位每个技能：`{raw, requirement(must/nice), cue(命中的线索词)}`——让"必备/加分"判定下沉到**技能级**，不再整行一刀切（修 P4 的 LLM 侧）。
- 规则降级模式：保留现有逻辑，但把"项目/经历段命中的技能"标 `project/work`、"技能栏命中"标 `listed`（修 P5 的规则侧），并把 `_classify_lines` 从"整行"改为"技能附近窗口"判定 must/nice（修 P4 的规则侧）。

### ② 受控归一层（规则查表 + 大模型补缺，钉进活表）

**现状（P2/P3 案发现场）**：`data/skills.py` 用"别名→单一 key"硬替换，且语义错误（`redis/mongodb` 挂在 `关系型数据库`，`fastapi/flask/django` 挂在 `python`）。无"关系"概念，只有揉平；揉平在抽取之后，故配密钥也错。

**改造分两步：**

**(a) 本体重构——用"关系"取代"替换"。** `data/skills.py` 拆出独立节点并加关系字段：
- 新增独立节点：`fastapi / flask / django / redis / mongodb …`，各有正确 `category`（Redis→"数据库/缓存"，不再是关系型）。
- 节点加 `implies: [key,…]`：如 `fastapi.implies=[python]`、`django.implies=[python]`、`redis` 不 imply `sql`。
- 语义：简历"会 FastAPI" ⇒ 满足岗位要求的 `python`（视为强证据），但 **FastAPI 这个技能本身保留**，不被抹成 Python。
- `match_skills` / `normalize_terms` 改为输出规范节点 + 其 `implies` 闭包，供③评分用。

**(b) 活表——大模型判断 + 钉死复用。** 新增持久化「归一表」（DB 表 `skill_norm`），承载本体之外、由大模型学习到的映射：
- 流程：抽取出原始词 → ②先查【本体 seed】→ 再查【归一表】→ 都没有(新技能)则**调大模型**判断它该归到哪个规范节点 / 或建为新独立节点 + 给 `implies` 关系 → **写入归一表钉死** → 之后同词直接查表。
- 字段：`{raw_term, canonical_key, implies(json), category, source(seed|llm|review), status(active|deprecated), created_at}`。
- 大模型在这里是这层的**大脑**，归一表是它**结晶下来的记忆**；套用永远是查表（确定性），仅"遇到没见过的词"才花一次大模型。
- 规则降级模式：无大模型时，新词走现有 free-form 兜底（`category=其他`），不阻断。

### ③ 评分层（规则·重做）

**现状（P1 案发现场）**：`gap_analysis._score`：`must_score = strong_have / len(must_agg)`（必备数量稀释）、弱证据固定 0.5、`final=0.8*must+0.2*nice`。

**重做要点：**
- **吃①的证据强度**：强弱来自抽取层的 `strength`，不再用 `_project_skill_keys` 重新 `match_skills` 猜（删除/弱化该函数）。`project > work > listed` 权重拉开。
- **吃②的关系**：岗位 `python` 必备，若简历命中 `python` 或任一 `implies python` 的技能（FastAPI 等）即算满足，证据强度取该实现技能的强度。
- **去稀释**：匹配度不再是"命中数÷必备总数"的机械比。改为按**关键必备的覆盖与证据质量**加权，并对"必备项很多"做合理归一（如按优先级加权、低优先必备不等额拉低分母）。具体权重曲线在实现时用代表性样例标定（见验收）。
- 输出不变契约：仍产出 `match_score / must_have_gaps / nice_to_have_gaps / possessed`，但每条带更扎实的 `evidence` 与 `strength`，供前端展示与④复核。

### ④ 复核层（大模型，握对话 + 联网数据）

**落点**：`agent.py` 的 `analyze_match` 工具在 `pipeline` 产出报告、yield `report` 之后，**回到对话循环**时增加一次"复核"。复核者就是对话里那个大模型（天然握全对话 + 之前 `web_search` 的最新行业数据）。

- 输入：③的权威分 + 全部缺口/已具备 + 每条 `evidence/strength` + ②用到的归一映射。
- 任务：分数合理吗？有没有把强证据当弱？归一有没有揉错（把不相关技能算命中）？结合最新行业数据，缺口判断对不对？
- 产出 A：**「AI 复核意见」**（自然语言，作为对话里的一段话给用户；**不改权威分**）。
- 产出 B：**结构化「问题记录」**落库（新 DB 表 `scoring_issue`：`{analysis_run_id, issue_type, skill_key, evidence, suggestion, status(open|applied|dismissed), created_at}`）。落库走新增小工具或结构化输出，复用现有 usage/归属上下文。

### ⑤ 规则优化（受控，非运行时）【默认受控，待 review 确认】

- `scoring_issue` 的 open 记录累积 → 评审（人工/半自动）→ 据建议**修改②的归一表或③的评分参数**（代码/表层）→ 过测试 → 合入 → 标 `applied`。
- **受控**＝不在运行时静默改分/改规则；权威分永远当场可复现。④甚至可在 `suggestion` 里给出具体改法，但要经测试 + 确认才生效。
- 这是默认取舍：若你要更自动（如复核直接写归一表纠错），可在 review 时调整——但会牺牲部分可复现性，需明确接受。

---

## 3. 可复现策略（关键，面试硬点）

"可复现" = **同一份存档、归一表稳定之后，重跑得同分**。三道闸：
1. **抽取结果持久化**：复用现有 `Resume.structured`（`pipeline` 已支持 `resume_structured` 跳过重解析）。
2. **归一钉死**：新词第一次由大模型判断后写入 `skill_norm`，之后查表；一次性判断、永久钉住（同①抽取缓存套路）。
3. **评分纯规则**：③给定上面两者后是确定性函数。

大模型出现在"第一次看见某新词"与"回头复核"两处，都不进入权威分数的运算。

## 4. 两种模式各层在不在

| 层 | 大模型模式 | 规则降级模式 |
|---|---|---|
| ① 抽取 | 大模型抽取 + 证据强度 | 规则抽取（已修 P4/P5 的规则侧） |
| ② 归一 | 查表 + 新词大模型补 | 查表 + free-form 兜底 |
| ③ 评分 | ✅ 重做（共享） | ✅ 重做（共享） |
| ④ 复核 | ✅ 有 | ✖ 无（无大模型） |
| ⑤ 优化 | ✅ 有 | ✖ 无 |

②③ 共享是关键：P1–P3 这类"配不配密钥都有"的病，根治在共享的②③层。

## 6. 兼容与迁移

- **对外契约不破**：`AnalysisRun.result` 的字段形状（`match_score/skill_gap/possessed/...`）保持，前端无需改；只是值更准、`evidence/strength` 更扎实。
- **新增存储**：`skill_norm`（归一活表）、`scoring_issue`（复核问题记录）两张表，经 `db_guard` 启动期校验。
- **本体扩充**：`data/skills.py` 由 ~30 节点扩为更全 + 关系字段；旧 key 尽量保留以兼容已存 roadmap/Task 的 `skill_key`。
- **旧分数不回填**：历史 `AnalysisRun` 不重算（避免动历史数据）；新分只对新分析生效。

## 8. 风险与取舍

- **归一表漂移/重复**：大模型补的条目可能前后不一致或与 seed 撞车 → 靠 ⑤ 评审 + `status` 去重维护；写入时先查 seed/表避免重复。
- **首次新词的判断成本**：每个从未见过的技能多一次大模型调用；但只首次、且可批量；命中表后零成本。
- **复核噪声**：④可能对正确分数也"挑刺" → 提示约束为"只在发现具体问题时记录，否则简短确认"，避免每次都长篇质疑。
- **规则模式上限**：无大模型时①②退化，质量仍逊于大模型模式——保持现有"演示走大模型、规则仅离线兜底"的口径。
