# OfferPilot 流程与匹配重构（父任务）

## Goal

总盘子，本身不直接写代码。统辖两条**相互正交、可各自独立验收**的改造线，并负责它们的总目标、任务地图、跨子任务验收与最终整合评审。

## 背景与动机

经过对当前代码的完整测绘（2026-06-22，6 路并行通读），确认两件事：

1. **当前是"以对话为中心、可反复新建多会话"的信息架构**，与用户真正想要的"一个学习周期一波流"心智不符；而后端 `JourneyState` 本就是"单用户单一 active 周期"，错位的恰是前端。
2. **匹配诊断（产品的"触发期"门面、README 自称全库最强）实际不准**：评分公式失真 + 技能库死板且有语义错误，且部分错误在"抽词之后的归一层"，**配不配大模型都错**。

两条线动的代码层不同（一个是流程外壳，一个是分析内核），几乎不重叠，故拆成两个独立子任务、各自验收。

## 任务地图

| 子任务 | 目录 | 改的层 | 执行顺序 |
|---|---|---|---|
| ① 匹配准确度修复 | `06-22-match-accuracy` | 分析内核（打分/技能库/抽取归一） | **先做** |
| ② 一波流·周期制改造 | `06-22-one-flow-journey` | 流程外壳（信息架构/状态机/双对话） | 后做 |

- **正交性**：在"一波流"眼里，"做匹配分析"是个被调用的黑盒；它内部用什么算法对流程改造透明。两条线代码基本不碰头。
- **父子非依赖锁**：顺序仅是组织约定，不是工具强制。先①后②写在此处。
- 父任务一般不承载实现工作；如需直接改动，再单列。

## 跨子任务验收标准（两条线都完成后才检查）

- [ ] 端到端走一遍：开场对话 → 匹配诊断（数字**像样、可解释**）→ 反问偏好 → 出计划 → 点【开始学习周期】→ 进入旅程执行期 → 旅程内对话能做面经/掌握度 → 结束周期进往期回顾。整条链路跑通无断裂。
- [ ] 匹配诊断在"一波流"的开场段调用时，输出与子任务①修复后一致（没有因流程改造而回退到旧算法）。
- [ ] 两条线各自的验收标准（见各子任务 PRD）均已单独通过。
- [ ] 演示口径自洽：不再出现"过得去的简历只有 17%""Redis 凭空消失"这类会被面试追问到哑口的现象。

## 约束

- 遵循 AGENTS.md：全中文、保持现有技术栈、改动写入 `docs/修改日志/`、提交按 `Agent-*` 尾注规范。
- "看懂+砍"模式：优先简化与接通，不新增"焊死未通电"的预留结构；能用现成数据/规则就不引入不可复现的新依赖。
- 单租户现状（设备 id = local）不在本轮做真账号体系。
