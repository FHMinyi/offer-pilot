## 物化契约（务必稳定 —— 与 materialize.py 顶部 docstring 一致）
1. **拆分**：每个 WeekItem 内 `tasks[i]`→`kind="learn"`、`deliverables[i]`→`kind="deliverable"`、`interview_focus[i]`→`kind="interview"`。
2. **order_index**：在「该 run 该 week」内**跨 kind 从 0 连续编号**（先 learn、再 deliverable、后 interview），作物化幂等业务键 + 周内稳定排序。
3. **字段**：`title`=裸字符串；`skill_key`=该 week `focus_skills[0]`（roadmap 不暴露技能 key，故存其名、截断 64；无则 `""`）；`weight=1`；`planned_date = (journey.start_date or run.created_at.date()) + (week-1)*7`。
4. **幂等 merge-upsert**（业务键 `(analysis_run_id, week, order_index)`）：
   - 命中 → 更新展示性字段(title/kind/skill_key/planned_date)，**保留用户进度**(`status=done/doing` 与 `done_at`)；若旧行是系统软删的 `skipped` 且本次又出现在 roadmap，则**复活为 todo**（skipped 是系统软删、非用户进度）。
   - 未命中 → 新建 `status="todo"`。
   - roadmap 删去且非 done 的旧 Task → 标 `status="skipped"`（软删不物删，保证 `CheckIn.completed_task_ids` 引用 Task.id 永不悬空）。

## 里程碑二接缝（显式记账，本期不实现）
- **位置对账非语义对账**：本契约只覆盖「**同一 run_id** 重跑 generate_plan」（调周数/换偏好）。`order_index` 是位置对账——weeks 变化会令 roadmap 重新分配技能，同一 `(week,order_index)` 内容可能换成别的技能，此时已 done 行按位置保留 done（里程碑一接受的取舍）。
- **跨 run 迁移映射（必入档）**：「**换 JD / 重新 analyze 产生新 run_id**」时旧 run 的 Task 如何迁移到新 run，本期**不实现**，留作里程碑二一等待办：按 `skill_key/title` 模糊对齐 + 平移已 done / 被 CheckIn 引用的 Task。

## 原因
roadmap 重生成一致性是里程碑一**最大风险**（演示中最易触发的「调周数/换 JD 重跑」）。先把物化做成稳定主键 + 幂等 merge-upsert 的纯函数、并配齐重跑/缩减/复活用例，C4 接主链时只需在 commit 后旁路 try 调用，风险已被单测前置锁死。结构化复用接缝省 LLM 重解析成本，并为里程碑二增量更新铺路。

## 抉择
- **C2 不接任何路由/主链**：服务层独立可测、可回滚（删两文件 + 移除 `resume_structured` 默认参即回原解析），把主链改造的风险隔离到 C4。
- **skipped 复活为 todo**（对方案「保留 status」的细化）：方案原文「命中保留 status」针对用户进度(done/doing)；skipped 是系统软删，若任务重回 roadmap 仍保留 skipped 会变僵尸隐藏行，故复活为 todo。done/doing 仍严格保留。已在契约与 docstring 显式标注。
- **`skill_key` 存技能名而非 key**：roadmap WeekItem 只暴露 `focus_skills`（名），不含 key；本期存名作提示字段（里程碑二可细化映射），截断 64 防越界。
- **materialize/ensure_journey 各自 commit+refresh**：便于 C2 独立测试落库；C4 主链 `db.commit()` 后旁路调用，二者各提交自身新增，单用户 SQLite 无并发问题。
