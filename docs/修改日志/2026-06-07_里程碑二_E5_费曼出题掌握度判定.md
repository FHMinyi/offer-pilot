## 原因

点勾完成是「自我声明无校验」，导致进度条造假、面试翻车。F1 是事后补救（面试挂了才发现没学透），E5 把校验前移到学习环节：用费曼复述 / AI 出题暴露「以为懂了其实没懂」，并把「完成(done)」升级为更高荣誉「真掌握(mastered ⭐)」。核心价值在于把 F1 的「输入→AI 分析→结构化盲区→权重回灌」抽象成统一的**学习闭环引擎**，费曼判定是它的第二个实例——这比堆功能更能体现架构复用与抽象。

## 抉择

- **软增值而非硬门禁**：仍可点勾，mastered 是拉力（徽章+真掌握率），不强制。避免把陪跑助手做成烦人考官。
- **双层状态用独立 `mastery` 列**，不扩 `TaskStatus`：done/mastered 是正交两轴，独立列向后兼容、不污染既有状态机，并为 v2「间隔重复 reviewing」预留。
- **持久化 `MasteryCheck` 表**而非瞬态：与 F1 `InterviewLog` 同构，判定历史可回看，是间隔重复的地基，也是工程叙事。
- **AI 当教练不当法官**：判定给建设性反馈+追问+缺口，`passed` 只升级不卡死；用户对「我已掌握」有最终决定权，AI 误判/缺席都不阻断。
- **事务比 F1 更稳一档**：判定核心结果（本体+升级）与回灌副作用拆成两段提交，回灌失败不丢判定结果。
- **db_guard 列级检测**：加列是本期唯一「上线即炸」风险（SQLite 不 ALTER 旧表），把它前移到启动告警根治此类问题。

## 已知局限 / 后续

- **数据迁移**：现有 SQLite 开发库需删库重建（`rm offerpilot.db && seed_demo.py`）或手动 `ALTER TABLE tasks ADD COLUMN`；测试库每次新建无此问题。引入 Alembic 后由迁移承接。
- **缺口回灌广度**：主题高度集中的 roadmap（如 demo 全是 Agent 主题）下，一个 high 缺口可能命中较多任务（实测 16 条）——属预期（盲区确实关联多任务），未来可加单次回灌任务数上限。
- **tone 复用未接**：MasteryDrawer 已留 `tone` prop 接口，但判定请求暂未带 tone（E3 语气滑块影响判定措辞为 v2）。
