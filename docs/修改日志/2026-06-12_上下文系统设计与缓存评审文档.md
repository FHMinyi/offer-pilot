## 原因

- 用户连续追问"上下文系统如何设计""会不会导致缓存命中极低"，结论值得固化留档：
  跨轮 prompt cache 命中率结构性归零（只剩固定指令头），叠加历史全量重传的 O(n²)
  输入增长，长对话约九成本可由缓存吸收的 input 成本目前全额支付。
- 修复刀口极小（时间戳挪位/降粒度 + 段序重排 + cache_control 断点），且 usage 子系统
  的 input_hit/input_miss 观测已就绪，修完当天可在面板验证——适合作为下一个小里程碑。

## 抉择

- **三个缓存问题分列而非合并**：它们相互独立（修任何一个都不自动解决另两个）、
  效果叠加，合并陈述会掩盖"R1 单独就能吃到大头收益"的优先级信息。
- **历史保真转录（R6）显式后置**：它是唯一彻底解决 P2 的方案，但要动前后端协议与
  存储；先吃 R1-R3 的九成收益符合演示项目定位。
- **评审方式沿用多智能体读码 + 主会话核证**：4 个并行读码智能体建立带 file:line 的
  事实清单，缓存结论（机制层面）由主会话直接读 agent.py/llm.py/usage.py 核证；
  未做真实 provider 上的 A/B 实测，已在文档 Agent-Limitation 注明。
- **与既有文档交叉引用而非重复**：P4 与《前端性能审查_2026-06-12》的"每轮全量 POST"
  是同一现象的网络/账单两面；P6 与差距分析的"灵魂≈0"同根，均只引用不重述。
