# 修改日志 · 2026-06-12（修复：输入区点击无法聚焦——ChatView 拆分引入的回归）

> ChatView 拆分重构（同日 C4-C6）后，对话页输入框点击后焦点立即落回 `<body>`，
> composer 永远无法展开，无法输入。无头浏览器（Playwright + chromium）复现、
> 定位、修复并回归验证。

## 修改内容

- `frontend/src/components/chat/ChatComposer.vue` —
  - template：展开区（chips/各面板的 `<template v-if="composerExpanded">`）与常驻的
    `<ModelSettingsPanel>` 一起包进新的常驻容器 `<div class="composer__panels">`；
  - style：`.composer__panels { display: contents }`（不产生盒子，子节点仍直接作为
    `composer__box` 的 flex 项参与布局，gap/视觉零变化）。

## 原因（根因链）

1. 拆分时 `ModelSettingsPanel` 从「展开区内部的 `<div v-if>`」改为「常驻组件实例 +
   `:open` 控制渲染」（为了拉取状态跨开合留存），位置夹在展开区 fragment 与
   `composer__row`（输入行）之间——父级子节点结构与重构前不同。
2. 点击输入框 → focusin → `composerExpanded` 翻真 → Vue 重排 `composer__box` 子节点时
   把 `composer__row` **移出 DOM 再插回**（MutationObserver 实测 `removed:composer__row`，
   节点同一）。浏览器对「被移动的聚焦元素」会直接丢焦点。
3. 失焦 → `onComposerFocusOut` 的 setTimeout 判定焦点不在容器内 → 折叠回去。
   净效果：focus 后 2ms 即 blur，永远无法展开。
4. 对照实验：worktree 起重构前版本（efa8da2）跑同一脚本——聚焦保持、无节点移除，
   坐实为本次拆分引入的回归。

## 抉择

- **修结构而非修焦点**：备选方案有「失焦判定中检测刚展开则重新 focus()」（治标，
  时序脆弱）与「把 ModelSettingsPanel 移回 fragment 内部」（需把拉取状态提升到模块级，
  改动面更大）。选择常驻容器方案：DOM 插拔被限制在容器内部，输入行在父级子节点列表中
  的位置永远稳定，根因消除且保留「组件实例常驻」的既有收益。
- **`display: contents`**：容器只为稳定 DOM 结构存在，不参与布局——展开态的 gap、
  子项顺序与未包裹时逐像素一致；折叠态容器内仅剩注释节点，不触发父级 gap。
- 注释在 template 与 style 两处互相引用说明约束（「为什么这个 div 不能删」属代码
  本身无法表达的约束）。
