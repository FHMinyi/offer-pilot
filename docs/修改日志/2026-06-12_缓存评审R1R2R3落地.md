## 原因

- 落地《上下文系统设计与缓存评审_2026-06-12》的最小组合 R1+R2+R3：秒级时间戳插在
  system prompt 中部导致每轮前缀缓存在中部折断，简历/JD 材料段与对话历史永远 miss；
  anthropic 路径无 cache_control 则是 0 命中。三处合计百行内，吃掉评审估算的
  约九成可缓存浪费。

## 抉择

- **降粒度在后端单点实施，前端不动**：前端仍发秒级 `client_time`，由 `_now_str`
  统一归一——后端不能信任客户端格式，单点收口比双端约定可靠；正则对未知格式
  no-op（缓存受损但功能正确），无解析失败风险。
- **时间挪到 prompt 末尾而非移入最后一条 user 消息**：后者对历史前缀更优，但要
  动消息语义且收益被 P2（历史有损重投影）封顶——缓存命中区本就止于第一个工具轮，
  小时粒度 + 置尾已拿到同等命中区，改动小一个量级。
- **易变项尾部内 tone 在时间之前**：时间每小时必变（确定性），tone 仅用户拨动
  滑块时变（偶发）——更易变者更靠后，折断面最小。
- **断点打在 anthropic_stream 而非 `_messages_to_anthropic`**：转换器保持纯函数
  （既有转换测试不受扰动），断点作为请求装饰在发送点叠加；`_call_anthropic`
  一次性调用（简历/JD 解析）prompt 短且无跨请求复用场景，不打断点（缓存写入
  有 1.25× 溢价，无复用则纯亏）。
- **R6（历史保真转录）维持后置**：本次不动前后端消息协议，P2 的分叉点照旧，
  属评审中已显式记账的遗留。
