## 原因

- 系统提示里的全局「当前时间」是缓存前缀里的可变字节，每小时变一次就冲掉其后全部历史缓存
  （R1 的小时粒度只是权宜）。**每条消息自带时间戳**发出即不可变、是 append-only 前缀，
  零缓存代价；且让模型能做跨轮/跨天时间推理，契合「数月级陪伴」定位（产品真功能，非纯优化）。
- 语气在系统前缀里时，拨一次滑块就腰斩其后历史缓存。只有把它移到历史**之后**的尾注才能保住
  历史——系统提示内部怎么重排都不行（系统提示恒为历史的前缀）。

## 抉择

- **取代 R1、取代 R2 的语气放置、保留并收紧 R3**：时间不再进系统提示（R1 的可变行删除）；
  语气从系统尾部移到消息尾注（R2 的「从稳到变」内部排序对语气已无意义）；R3 的 anthropic
  断点保留，但末消息断点打在真实用户块、尾注追加在其后，确保改语气不动前缀。
- **助手时间 = `onDone`（全部回复完成）那一刻**：按用户要求，流式中途与工具调用不计时；
  abort/error 不盖（`onDone` 只在正常 `done` 事件触发）。
- **时间前缀后端单点收口**：前端只采集本地字符串，拼 `【时间】` 由后端统一做——不信任客户端
  格式；最后一条用户消息无 time 时用 `client_time` 兜底，保住「现在」锚点（老会话/老客户端兼容）。
- **尾注绝不进历史/不持久化**：每轮用当前 tone 现生成，避免旧语气残留进被缓存的前缀。
- **不可用 anthropic「中途 system 消息」**：该能力仅 Opus 4.8 起支持，本项目默认 sonnet-4-6 会 400；
  故走「末消息追加 text block」这条跨档可用的路径。
- **素材外移留作下一刀**：唯一要动持久化 schema + 前端事件模型 + 换简历语义的部分，单独验证，
  本次不碰（仓库无 CI/lint 兜底，不把高风险项捆进来）。
