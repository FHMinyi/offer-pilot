# 修改日志 · 2026-06-10（token 用量统计子系统：落库 + 全路径 + 缓存命中分类 + 独立图表页）

把目前被丢弃的 LLM 响应 `usage` 捞起来落库，做成一个**给开发者自用的「提示词调优仪表盘」**：
对话每条助手消息气泡下方显示本轮 token 用量；一个独立页面 `/usage` 用条形图按
天(每小时)/周(每天)/月(每天) 展示 **输入命中缓存 / 输入未命中缓存 / 输出** 三类 token，
可**按模型、按功能拆分**，并提供**缓存命中率视图**——目标是看清哪条功能/模型缓存命中差，
据此把稳定前缀前置、最大化 prefix cache 命中。覆盖全部 6 条真实 LLM 路径。只统计 token、不算钱。

## 原因
- 项目对 API 成本/缓存早有意识（模型分档、`resume_structured` 复用省调用），但 token 用量从未
  对用户可见。该子系统把它落库 + 可视化，直接服务「开发时调提示词、提升缓存命中」的诉求。
- 缓存命中率随时间/功能的对比是核心抓手：命中率低 = 前缀不稳定，提示把固定指令前置；命中率
  视图能验证这类改动前后的效果。
- 覆盖全部 6 条真实 LLM 路径（`chat/resume/jd/optimize/blindspot/mastery`），按模型 + 按功能
  双维拆分，才能定位「最该优化」的那一项。

## 抉择
- **独立新表 `token_usages` 而非给旧表加列**：`db_guard` 的 `create_all` 只建缺失表、绝不 ALTER
  旧表——给旧表加列会在已有库静默缺列。新表既零告警，又天然满足跨会话历史总计。
- **捕获走 contextvars、零 service 签名改动**：6 处 LLM 调用散落 5 文件，加参要污染
  `parse_resume/parse_jd/suggest/extract_blind_spots/judge_*` 全链路；复刻既有 BYO override 的
  contextvars 范式（`copy_context`+`ctx.run` 已验证流式全程可见），最小侵入。
- **时区分桶在后端按 `tz_offset` 落本地桶 + 补零**：与本仓 `localTodayIso`「前端传本地、后端对齐」
  一脉相承；规避 SQLite `strftime` 对 `+00:00` 后缀的不稳定。单用户数据量小，Python 内存分桶足够。
  DST/半时区用固定偏移近似（演示可接受）。
- **本会话累计纯前端、DB `conversation_id` 对话路径填 None**：流式期 conversation_id 尚未落库
  （前端事后保存），故不引入 `ChatRequest.session_id`；跨会话总计由 DB 权威，会话累计前端 reduce。
- **`path` 枚举只有 6 个，剔除 `gap`/`roadmap`**：实证 `gap_analysis.analyze`、`roadmap.generate`
  是纯规则、不调 LLM，不为其打标签（未来若改用 LLM 再加 path）。
- **三类 token 端到端统一 `input_hit/input_miss/output`、命中率前端派生**：DB 列=API=SSE=前端类型
  单一口径，不返回 `hit_rate`/`total` 冗余字段（除零归一在前端显示「—」），避免命名翻译层与序列化分歧。
- **统计端点强制按设备过滤**（不走 `ownership.scope_to_user` 放行）：统计语义即「我的用量」，演示期
  也隔离，与其余路由「里程碑三才收紧」节奏不同，是有意为之。
- **图表手写 SVG、不引库**：项目零图表依赖、`ScoreRing.vue` 已是先例；多序列 tokens 用
  small-multiples（避免并排条颜色冲突、窄屏不可读），命中率用单坐标系多折线（直接比高低）。
- **`stream_options` 兼容兜底**：不认的兼容端点（vLLM/Ollama）在重试分支里一并去掉再请求，
  usage 取不到则记 0、不污染图表，对话功能不受影响。
