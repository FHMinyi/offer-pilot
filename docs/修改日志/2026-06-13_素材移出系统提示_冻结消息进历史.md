## 原因

- 简历/JD 正文（占系统提示 ~30-50%）嵌在系统提示这个前缀里，**中途新增/更换素材**（典型
  "先打招呼、后传简历 JD"）会改变前缀、一次性冲掉此前全部历史缓存。这是时间/语气之外仅剩的
  高频前缀失效源。唯一根治：把素材搬进 append-only 的消息历史、在引入它的那一轮冻结。

## 抉择

- **换简历/删 JD 的语义 = 重新冻结 + 时间线**：每次发送比较当前素材与上次冻结，变了就把当前完整
  快照冻结到本轮。旧的冻结消息**永不回改**（回改即破坏缓存、且不诚实）。模型看到「先 A、后改 B」
  的真实时间线，分析用 context 最新版；最新那条冻结消息恒等于 context，二者不再「错位」。
- **context 与冻结消息并存、各司其职**：context 做分析权威源（最新），冻结消息做模型会话视图（历史）。
  非冗余浪费——素材 token 只是从系统提示位置挪到冻结消息位置，且分析读 context 不额外占 token。
- **界面只显紧凑标记、不显正文**：素材正文进模型不进界面，避免 4000 字塞满气泡。
- **`attachLabel` 提到 chatModel.ts 共用**：ChatView/ConversationView 不复制（守仓库复用纪律）。
- **边界显式记账**：① 全部移除素材 → v1 不附「已清空」标记（罕见）；② 老会话续聊 → context 恢复
  但 turns 无 attached，首次发送 `pendingAttachment` 重新冻结当前素材（功能正确，时间线显示为"现在附"）；
  ③ role/weeks 仍在系统提示（体量极小，留作可选小尾巴）；④ 存储轻微冗余（context + turns 各一份）。
