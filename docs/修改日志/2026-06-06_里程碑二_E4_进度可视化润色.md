## 审查后修订（对抗式 review 工作流：18 条发现 → 核验确认 12 条）
跑了一个多智能体审查工作流（4 维度独立审查 + 每条发现独立怀疑者对抗核验），据确认项修订：
- **跨时区口径（major）**：`/api/progress` 新增可选 `today` 查询参数（客户端本地自然日），
  前端 `client.ts getProgress()` 统一传 `localTodayIso()`，让 `recent_days`/`checked_in_today`/
  `streak`/`current_week` 都以浏览器「今天」为锚点，与打卡 date（本地日）及 checkins/replan 同口径，
  修掉「服务器时区≠用户时区且跨日临界」时整体错位一天的隐患（连既有 streak 一并修正）。
- **节奏洞察用实时值（minor）**：`insight` 的剩余条数改用实时 `total-done`（不再用 `signals` 旧快照），
  「全部完成」无需重排过即给正反馈；仅「上次结算顺延 N 条」如实保留快照语义。
- **删死代码（minor）**：移除从未生效的 `weekbar__scale` 透明元素 + `maxWeekTotal` + 误导注释，
  周条明确只表达「本周完成度」，工作量见右侧 done/total 计数。
- **reduced-motion 全覆盖（minor）**：媒体查询补 `weekbar__fill`/`streak__cell` 的 `transition: none`，
  并给共享组件 `ScoreRing` 加同样守卫——淡入/数字滚动/脉冲/进度过渡/环形扫动全部可关。
- **token 一致性（nit）**：两处硬编码 `#fff` 改 `var(--text-on-brand)`。
- **测试强化（nit）**：`recent_days` 用例加「窗口外打卡不计入 + 恰为连续 7 日序列」断言；
  新增 `test_progress_accepts_client_today`（传入固定客户端 today 驱动锚点）。
- **保留不改**：入场 `nth-child` 错峰延迟会随 v-if 组合轻微漂移（纯视觉、仍渐入），评为可接受 nit。

## 原因
E4 是简历项目里「便宜加分」的可视化润色：进度看板是 demo 里最直观的「正反馈」画面。坚持天数从
单个数字升级为 7 天热力条，相对完成度从静态条升级为带动效、带阶段连线的看板，并把 E1 再规划的
结果（顺延了几条 / 剩余几条）显性带到看板——让「动态再规划」这条工程故事在 UI 上看得见。

## 抉择
- **后端只加 `recent_days` 7 单元、不存表**：复用 progress 已查出的打卡日期集合即时生成，
  与「lifespan 仅 init_db、无 cron」一致；逐日精确（非由 streak 推断），避免空档被误判为连续。
- **热力配色用 `--warning` 实底而非自造橙色渐变**：与侧栏 🔥streak 角标同色、守设计 token，
  避免引入硬编码颜色的「惊喜」（延续 C5/E1 的取舍）。
- **节奏洞察读 `signals` 且仅在重排后显示**：`signals` 仅由 `replan_journey` 写入，
  以 `last_replanned_at` 为门，避免空 `{}` 被误显示为「全部完成」。
- **动效一律 `prefers-reduced-motion` 可关**：无障碍底线；数字滚动/脉冲/淡入全部受控。
- **节奏文案守 B7**：只说「剩余/顺延 N 条任务」「节奏稳健」这类相对进度，绝不报 Offer 概率。
