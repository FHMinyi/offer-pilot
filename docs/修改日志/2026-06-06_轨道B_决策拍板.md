### 8 个决策最终选择
- **B1 护城河方向 = 数据 + 体验并重**（用户选择；与"数据为主"推荐不同，MVP 焦点更宽、成本更高，已尊重）。
- **B2 楔子人群 = 在校生**（获客/口碑层 + 校园裂变，先验证留存，付费验证留后续）。
- **B3 五阶段建模 = 多维并存信号**（进度健康度/情绪/面试活跃/中断天数各自独立，stage 派生）。
- **B4 终局设计 = 多终态**（含暂停/未达成/主动退出）。
- **B5 人设引擎 = 单人设 + 语气滑块，预留三人设数据结构**。
- **B6 真实环境感知 = 联网公开检索近似 + 用户粘贴 JD 并用**，不做平台抓取（推迟 M3）。
- **B7 Offer 概率话术 = 相对进度/完成度**（不报概率，判决类数字隐藏/只给趋势）。
- **B8 地基技术路线 = Thin-slice**（焊 user_id 接缝、暂不做真账号；不引 Alembic；锚定 AnalysisRun）。

### 原因
轨道 B 是 C 轨建表前必须定的设计前提（行动计划依赖图：A、B → C）。逐个拍板后，C1 建 `JourneyState/Task/CheckIn` 三表才有确定的字段口径。

### 抉择
- **B1 选"并重"而非推荐的"数据为主"**：用户判断情绪体验同样是核心。已在行动计划标注其代价（MVP 焦点更宽、需两头投），不强行收窄。
- **B3 用 `signals: JSON` 容器而非一堆固定列**：M1 多数信号由 Task/CheckIn 实时派生，少数显式存；JSON 容器对 M2 扩展信号零改表，符合 thin-slice「预留接缝」。
- **B4 用扩展 `status` 枚举承载多终态**而非新建状态机表：M1 单用户、终态有限，枚举足够；M2 再视需要拆分。
- **B5 存 `persona`+`tone` 两字段**而非单 `tone`：正是用户选的"预留三人设结构"，将来加命名人格零返工（对应 E3）。
- **B6/B7/B1 不改这三张新表**：B6 影响 JobPosting(已有 source_url)/联网路径，B7 影响 GET /progress 输出与前端，B1 是路线优先级，均不落在 M1 的三表 schema。
