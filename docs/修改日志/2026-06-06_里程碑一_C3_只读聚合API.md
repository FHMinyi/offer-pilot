## 原因
「先读后写」把零写入风险的聚合端点先打通，前端 C5 才有可渲染的进度/任务/旅程数据；streak 用惰性聚合而非建表/cron，严守「lifespan 仅 init_db」约束。

## 抉择
- **三个读端点统一走 `scope_to_user` 放行而非硬 `where(user_id==user_id)`**：满足验收「foo 与无头读到同一份数据」，并把多租户隔离收敛到 `ownership.scope_to_user` 单点，里程碑三一处启用即全站收紧（写路径 `ensure_journey`/`materialize` 仍按 user_id 归属，读放行是本期已知 `KNOWN-MULTI-TENANT-LEAK`）。
- **`done` 用 ORM 只读 property 而非 mapped 列**：避免与 `status` 双写不一致；序列化由 `from_attributes` 读取 property，前端拿到便利布尔，无需自行推导。
- **progress 排除 `skipped`、按旅程绑定 run 聚合**：completion_rate 只对当前活跃计划的有效任务计算，软删行不污染分母；空库/无旅程时回退零值与 `current_week=1`，前端空态可直接引导。
- **`current_week` 优先按 `start_date` 自然周推算**：与 streak 同口径（自然日），无 start_date 才回退旅程登记值；为里程碑二再规划写 `current_week` 留出读取点。
