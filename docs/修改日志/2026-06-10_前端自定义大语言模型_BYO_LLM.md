# 修改日志 · 2026-06-10（前端自定义大语言模型 BYO LLM + 判定类可选推理强度）

让用户在前端 UI 上**按会话自定义所用的大语言模型**（provider / 接入点 base_url / API Key /
三档模型），覆盖后端 `.env`，无需改配置、重启即可切换到任意 OpenAI/Anthropic 兼容服务；
并能从所填端点拉取可用模型列表供下拉（拉不到/任何时候都可手输）。顺带给「判定类」
（费曼判定 / 出题 / 答题判定）加可选推理强度。

## 修改内容

### 后端
- `backend/app/services/llm.py`（核心）— 新增 contextvars **覆盖层**：
  - `_override` ContextVar + `use_llm_override()`（单一 context 同步用，带 token 重置）
    + `set_override()`（流式路由用，仅 set、不重置）。
  - 按字段独立回退访问器 `_eff_provider/_eff_model/_eff_base_url/_eff_api_key`
    （+`_eff_model_server`）；`engine_name/llm_enabled/streaming_supported/model_for/
    complete_json/_call_openai/_call_anthropic/openai_stream/anthropic_stream` 全部改为
    经访问器取值。覆盖为空时行为与改前**完全一致**（回退 `.env`）。
  - `model_for(role)` 三档覆盖优先级：分档框 → 前端默认框 → 服务端分档 → 服务端默认。
  - `complete_json(..., effort=None)`：openai 注入 `reasoning_effort`、anthropic 注入
    `thinking` budget，兼容服务不认时去参重试兜底。
- `backend/app/schemas.py` — 新增 `LLMOverrideIn`（六字段，全空串默认）；`ChatRequest`
  加 `llm_override`；`FeynmanJudgeRequest/QuizGenerateRequest/QuizJudgeRequest` 各加
  `reasoning_effort: str = "medium"`。
- `backend/app/routers/chat.py` — 用 `copy_context()` 起一个专属 context，`set_override`
  设一次覆盖，再以 `ctx.run(next, inner)` 逐步驱动整条 SSE 生成器（详见「抉择」）。
- `backend/app/services/pipeline.py` — `_parse_inputs_streaming` 的两处 `ex.submit` 改为
  `ex.submit(contextvars.copy_context().run, parser, arg)`，把覆盖传播到并发解析子线程。
- `backend/app/routers/llm.py`（新）+ `main.py` — `POST /api/llm/models`：按 provider 复用
  `llm._build_client` 调上游 `/models`，成功 `{ok:true,models}`、异常 `{ok:false,error,models:[]}`
  （**绝不抛 5xx**，让前端优雅退回手输；Key 只走 POST body、不入日志）。
- `backend/app/services/mastery.py` + `routers/mastery.py` — 三个判定函数加 `effort="medium"`
  并透传到 `complete_json`；三端点把 `payload.reasoning_effort` 透传给 service。
- `backend/tests/test_llm_override.py`（新，14 项）— 分档覆盖/退出恢复、`complete_json`
  effort 注入与兜底、`/api/llm/models` 成功与异常不 5xx、pipeline 线程传播、以及
  **流式 chat 路由 ctx.run 覆盖跨 yield 传播的回归测试**。后端总计 114 项全绿。

## 原因
- 换模型原本必须改 `.env` 并重启（配置进程启动时 `@lru_cache` 一次性读取）；前端自定义
  让演示/试不同模型零成本，也契合项目「能演示 + 工程故事」的目标。
- 全项目 8 处调 LLM 分 3 档（简历/JD 解析用小模型，其余用主模型），故前端按三档自定义、
  留空回退，既灵活又保留省钱分档。
- 判定类（费曼/出题/答题）是「需要思考」的场景，加可选推理强度可提质；纯抽取的简历/JD
  解析不加（要稳定 JSON、加思考收益不明）。

## 抉择
- **contextvars 覆盖层而非到处传参**：`llm.py` 处处读模块级 `settings`，加一层按字段回退的
  访问器最小侵入；覆盖为空即完全等价旧行为。
- **流式路由用 `copy_context()` + `ctx.run(next, inner)` 而非 `with use_llm_override`**：
  Starlette 以线程池逐步迭代同步生成器，每次 `next()` 可能在不同线程/context——`with`/Token
  方案会在不同 context 间 `reset` 而报 “Token created in a different Context”，且覆盖无法
  可靠跨 yield 传播。显式专属 context 驱动每一步，覆盖全程可见、无需跨线程重置。
  （此坑实测先复现为 `test_chat` 报错，修复后补了专项回归测试。）
- **pipeline 子线程用 `copy_context().run`**：contextvars 默认不传播到 `ThreadPoolExecutor`
  工作线程，否则 chat 触发的分析里简历/JD 解析会忽略覆盖模型。每个 submit 各起一份 copy
  （同一 Context 不能被多线程并发 run）。
- **API Key 只存 `localStorage`、随请求发给本地后端、不写会话 DB**：单机自用/演示场景下
  足够，且避免把密钥落库；故未改 `ChatPersistContext` 与会话持久化。
- **模型选择用原生 `<input list=datalist>`**：天然「下拉 + 手输」二合一，端点 `/models`
  拉到就填充候选、拉不到仍可手输，零额外组件。
- **`/api/llm/models` 永不 5xx**：任何上游/SDK/鉴权异常都收敛为 `{ok:false,error}`，前端据此
  提示并退回手输，不打断使用。
