## 原因
里程碑一所有缺失能力（每日任务/打卡/进度/旅程状态）卡在同一块缺失地基——没有长期有状态的数据模型。本步落地三表即把这块地基补上，且全部 day-one 带 `user_id String(64)`（C0 接缝同形），后续 C2–C5 才有可物化/可读写/可勾选的承载。

## 抉择
- **三表口径严格照 §3.2 + B 轨决策**：`stage` 为派生展示标签、真值走 `signals(JSON)`（B3）；`status` 五终态（B4）；`persona`/`tone` 预留三人设（B5）；`profile_type` 预留四画像（B2）。这些字段本步只建列、暂不写业务逻辑，避免里程碑二/三再加列返工。
- **`Task.analysis_run_id` 必填、`journey_id` 可空**：锚定 AnalysisRun（不另建 Plan 主表），Task 是 `AnalysisRun.result.roadmap` 的可勾选投影；`journey_id` 可空容忍「先有 run 后补 journey」的物化时序。
- **唯一约束选 `(analysis_run_id, week, order_index)`**（裁决 C3）：单一稳定序，作 C2 物化幂等键 + 周内排序；`kind` 仍列存但不进唯一键，避免措辞/分类微调破坏对账。
- **不在本步接任何路由/主链**：C1 只让模型存在、表建出；旧链路不读新表，故 48 项全绿、零回归。读写在 C3/C4 分阶段叠加。
