## E1.1 · 任务排到「天」（周→日粒度）

### 修改内容
- `backend/app/services/materialize.py` — 物化时把每周任务按周内序 `order_index` 摊到本周 0..6 天：`day_offset = min(6, order_index*7 // 周内任务数)`，`planned_date = 周首日 + day_offset`（原本整周挤在首日）。更新物化契约 docstring。
- `backend/tests/test_materialize.py` — 改 `planned_date` 断言为「落在该周 7 天窗口内」；新增 `test_materialize_distributes_tasks_across_week_days` 验证一周任务摊到多天。

## E1.2 · 动态再规划引擎

### 修改内容
- `backend/app/services/replan.py`（新）— `replan_journey(db, journey, *, today, settle)`：
  - **顺延**：planned_date 早于今天且未完成的任务滚到今天起。
  - **重组**：未完成任务按 (week, order_index) 序，以每日容量 `cap=ceil(剩余/剩余天数)` 均匀重排到 [今天..计划末日]。
  - **降权**（仅 `settle=True`）：仍逾期未完成的任务 `weight-1`（下限 0），作"持续被顺延"至危信号。
  - 写 `JourneyState.signals`（progress_health / carried_over / remaining）+ `current_week` + `last_replanned_at`。done 任务不重排、不降权。
- `backend/app/routers/journey.py` — 加 `POST /api/journey/{id}/replan`（`ReplanRequest{settle,today}` → `ReplanOut{journey,tasks}`）。
- `backend/app/routers/checkins.py` — 打卡=每日结算：upsert 后自动对 active 旅程 `replan_journey(settle=True)`（try 包裹、失败不影响打卡）。
- `backend/app/schemas.py` — 加 `ReplanRequest`/`ReplanOut`；`JourneyOut` 补 `signals`/`last_replanned_at`。
- `backend/tests/test_replan.py`（新，6 用例）— 顺延+降权、重组容量分布、done 不动、同日幂等、HTTP 端点、打卡自动触发结算。

## 抉择
- **物化只设「初始」每日分布，replan 负责之后的调整**：避免两者抢写 `planned_date`；仅显式重新生成计划会重置回初始分布（已在 docstring 记账）。
- **降权用 `weight`（capped 0）作信号而非新增列**：复用 C 已建的可写字段，零改表；前端据 weight=0 弱化展示。同日二次结算无逾期 → 不再降权（天然幂等）。
- **打卡自动触发结算重排**：让"每日打卡"直接驱动计划自适应，是最自然的 demo；用 `today=打卡日`、try 包裹降级，不影响打卡本身。
- **重组按 (week,order_index) 序保留教学顺序**，仅按容量摊到日，不打乱 roadmap 的学习逻辑。
- **跨 run 任务迁移仍不做**（里程碑二后续/换 JD 场景），与 C2 记账一致。
