## 原因

评审判定 ChatView.vue 为全仓唯一高危热点：约 13 项职责聚集、占前端代码 32%、近期
每个特性（E3/BYO LLM/usage）都在此落点且改一个功能要横跨 script/template/style 三段；
气泡渲染在两视图间双份维护且已实际漂移。趁 E6 尚未在错误模式上继续生长，先拆。

## 抉择

- **report 块走作用域插槽而非 mode prop**：chip 的交互（focusReportPanel）与 960px 媒体
  切换留在父作用域，共享组件不感知两视图的呈现差异。
- **折叠态采用「共享 reactive 视图模型可 mutate」约定**而非 props/emits 单向流：流式期间
  useChatStream 也要写 reasoningOpen（自动展开/收尾全折叠），藏进组件私有状态会断链；
  约定已写进 AssistantBlocks 文件头。
- **序列化白名单投影一字不动**：后端 `conv.turns = payload.turns` 零校验，前端序列化函数
  就是事实 schema；vitest 用 JSON.stringify 断言瞬态字段（streaming/resultsOpen 等）不入库，
  并覆盖「最小存量 tool 块」「ok===undefined 往返」等老数据兜底。
- **keyframes 全局化并改名 op-***：Vue scoped 会给 keyframes 加 hash 并只改写同组件引用，
  跨组件引用会静默失效（spinner 静止无报错）——评审风险排查阶段识别的最隐蔽坑。
- **信号 watch 全留 ChatView**：ReportSidePanel 经 v-if 频繁卸载，清理逻辑放子组件生命周期
  会在「隐藏面板」时连带杀掉 SideNav 重开链路。
- **四条怪行为按「搬运红线」原样保留**（停止后落一次保存 / editTurn 不立即 persist /
  中断无 reasoning 置 noThinking / retry 不重置 usage）——重构不夹带行为变更，留待单独决策。
- **提交切三刀**（纯函数层→共享渲染器→composables+子组件）：每刀独立可 revert、各自绿灯。
