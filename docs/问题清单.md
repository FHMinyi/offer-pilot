## P1 匹配度评分偏低 / 失真　🟡待讨论　✅

**现象**：一份还过得去的简历，匹配度只有 17%，低得吓人。

**根因(代码在哪)**：
- `services/gap_analysis.py` `_score()`：弱证据只记 0.5 分、必备得分占 80% 权重；
  `必备得分 = 命中分 ÷ 必备技能数` —— 必备技能一多，分母就大，分数被稀释。
- 必备技能数容易「虚高」：规则模式 `jd_parser._classify_lines` 默认把行归「必备」；
  LLM 模式 `_parse_with_llm` 还把整个 `tech_stack` 并进必备。

**影响**：对一个求职工具，一个虚低的吓人分数是很糟的体验，会误导 / 打击用户。

**影响模式**：两种模式都有 —— 评分公式（`_score`）是共享的；「必备虚高」的来源因模式而异：
规则模式 `_classify_lines` 默认把行归必备、LLM 模式 `_parse_with_llm` 把整个 `tech_stack` 并进必备。

**面试角度**：「我发现评分失真 → 定位到公式偏严 + 分母膨胀 → 校准」是个能打的故事。

## P2 技能词典「折叠」过粗，还有语义错误　🔴待修　✅

**现象**：JD 里列的不同技能被揉成一个；有的还归错类。
- Python / FastAPI / Django → 全合并成一个「Python」（`data/skills.py` 的 python 别名含 fastapi/flask/django）
- SQL / Redis / MongoDB → 全合并成「关系型数据库」（`data/skills.py:206` 的别名表）
- ❗ **Redis、MongoDB 根本不是关系型数据库**，却被归进「关系型数据库」—— 语义错误。

**影响**：分不清「会 Python」和「专精 FastAPI」；技能信息被揉平，直接拉低分析质量。
词典只有 **30 个技能**，覆盖面也窄（词典外的技能规则模式直接看不见）。

**影响模式**：⚠️ **两种模式都有**。折叠发生在「归一化」那步、在抽词【之后】—— 即使配了 AI、
LLM 正确抽出了 FastAPI / Redis，下游照样把它们揉平。所以这不是「笨的离线模式专属」，是**真·产品问题**，更该修。

## P3 加分技能被「去重」误删（Redis 凭空消失）　🔴待修　✅

**现象**：JD 写「了解 Redis 者优先」，最终加分技能里 Redis 不见了。

**根因**：Redis 折叠成「关系型数据库」，而 SQL 已经让「关系型数据库」进了必备；
`jd_parser.parse_jd` 里有条「加分里与必备重复就删」的规则，把它当重复删掉了。

**影响**：加分项丢失，分析不完整。是 P2 的连带后果。

**影响模式**：⚠️ 两种模式都有（同 P2，去重也在归一化之后）。

---

## P4 规则模式「按整行」分必备 / 加分，太粗　🟡待讨论　✅

**现象**：必备 / 加分是按【整行的措辞】判的，不是按单个技能。

**根因**：`jd_parser._classify_lines` —— 一行里有「了解 / 优先」且没「必须 / 精通」才算加分，
否则整行算必备。于是「精通 Python，了解 Docker」这种混合行，**Docker 也会被算必备**。

**影响**：必备被高估（连带 P1 的分母膨胀）。仅规则模式；LLM 模式由模型判句义。

## P5 规则模式下简历的「项目经历」没被用于「强证据」判定　🔵待核实　❓

**现象**：演示里简历写了「用 Flask 写过博客系统」，但 Python 仍被算成「薄弱」(只 0.5 分)，
没拿到「项目实证」的满分。

**怀疑**：规则版简历解析（`resume_parser._parse_with_rules`）可能没把那句话整理进
`resume["projects"]`，导致 `gap_analysis._project_skill_keys` 抓不到项目里的技能。

**待办**：读 `resume_parser.py` 核实。（**还没读，先记着，别当定论。**）

## ⚪ 已知限制（设计取舍，不是 bug）

- **规则模式（无 AI 密钥）整体质量明显差**：解析糙、只认 30 词词典、把不同技能粗暴合并。
  这是「离线优雅降级」的代价，不是 bug —— 但**演示时别拿规则模式当门面**，要配密钥走 LLM 模式。
