# OfferPilot 实现说明

## v0.2.1：思考强度可调、对话注入时间、Anthropic 思考（2026-06）

- **思考过程可见与分档**：对话流新增 `reasoning` 事件，前端折叠展示模型思考；输入区「思考强度」6 档（关闭/低 low/中 medium/高 high/极高 xhigh/最高 max）。后端映射：openai 协议透传 `reasoning_effort`（API 上限 high，故 xhigh/max 截到 high）；anthropic 映射为扩展思考 `budget_tokens`。"关闭"档即使模型仍内部思考也不下发展示。不支持的模型自动丢弃参数重试，绝不报错。
- **注入当前时间**：前端把本地时间随请求 `client_time` 传入，后端写进系统提示，并要求联网检索使用当前时间范围，避免给出过时年份的信息。
- **分析进度可视化**：`run_analysis` 期间分步下发 `status`（解析简历/逐条解析 JD/缺口/建议/路线），前端在该工具卡上动态展示。
- **事件按真实顺序交错**：助手消息改为有序 blocks（text/reasoning/tool/report），渲染顺序即发生顺序。
- **取材健壮性修复**：`run_analysis` 工具可从附件 context 或模型在对话中提取的 `resume_text/jd_texts` 参数两处取材——用户把简历/JD 直接粘进对话框也能正确分析。
- **Markdown 渲染**：助手回复用 marked + DOMPurify 渲染。
- **Anthropic 对话路径**：新增非逐 token 的 anthropic Agent 单步（思考预算 + 工具调用 + openai↔anthropic 消息/工具格式转换）。说明：当前环境无 Anthropic Key，转换/映射有单测覆盖，但端到端未在真实 Claude 上联调，使用前建议自测。

## v0.2：转向流式对话式 Agent + 联网搜索（2026-06）

基于真实使用反馈，产品形态从「表单 + 静态结果页」演进为**流式对话式 LUI（Agent）**：

- **过程可见（流式）**：新增 `POST /api/chat/stream`（SSE）。一轮对话内，助手以增量文本流式输出，并把工具调用、运行分析、报告生成等过程实时下发（事件：status/delta/tool_call/tool_result/report/done/error）。解决「分析等待时黑屏、不知 AI 在做什么」的问题。
- **可控 Agent + 工具**：采用 OpenAI 协议的流式 function calling 编排（`services/agent.py`），工具有二：
  - `web_search`（`services/search.py`，Tavily）：检索某岗位方向**当前**常见技能/框架，弥补 JD 未写明的「行业默认」（例如 AI Agent 方向常默认 LangChain/LangGraph/RAG/向量库）。检索结果可经 `extra_jd` 并入分析，从而体现在缺口与路线里，且来源可追溯（标注「AI 联网整理」）。
  - `run_analysis`：复用既有 pipeline 产出结构化报告并持久化。
- **对话式界面**：前端重写为聊天式 LUI（上传/粘贴/追加 JD、流式消息、工具活动、内嵌报告卡、随时追问与调整重跑），报告渲染抽为可复用的 `AnalysisReport` 组件。
- **网络出口**：搜索出网显式走选定的 http(s) 代理并 `trust_env=False`，刻意绕开格式可能有误的 `ALL_PROXY/socks`；**只读取、不修改**任何代理环境变量。LLM 客户端同样对损坏的代理变量做自愈回退。
- **降级**：未配置 LLM（或非 openai 协议）时，对话退化为脚本化流程（收集简历/JD → 规则分析 → 出报告）；未配置 Tavily 时仅凭模型已有知识作答。两者都保证「可用」。

### 与 AGENTS.md 的关系

AGENTS.md 原则建议 MVP 阶段「先 Workflow、暂不优先 Agent 编排与搜索」。v0.2 引入 Agent + 联网是**产品负责人基于实际体验的主动决策**，并非偏离：底层结构化分析（解析/缺口/路线）仍是确定性可控的，Agent 只负责编排、解释与信息补全。后续可据此同步更新 AGENTS.md 的阶段约定。

# OfferPilot MVP 实现说明 v0.1

> 本文记录 MVP 第一版实现中相对于 PRD 草案做出的具体取舍，作为对 `OfferPilot_PRD_技术方案草案.md` 的实现补充（遵循 AGENTS.md：技术选型/范围变更需同步文档）。

## 范围

实现了 PRD Phase 1「单次分析闭环」的全部必做功能：简历上传/粘贴、JD 输入、岗位画像、技能缺口分析、简历/项目优化建议、按周学习路线、结果导出与历史记录。

暂缓：自动爬取 JD、面试模拟器、自动投递、大规模学习资源推荐（与 PRD §4.2 一致）。

## 关键技术取舍

### 1. 数据库默认用 SQLite（而非直接上 PostgreSQL）

- 原因：MVP 追求零配置、可一键跑通与易调试，符合 PRD §9.3「能放数据库里的，不要一开始就依赖更重的基础设施」。
- 做法：通过 `DATABASE_URL` 一行切换到 PostgreSQL，生产可平滑迁移。
- 结构化解析结果统一以 JSON 字段存储，暂不拆出 `ProjectExperience / SkillNode / EvidenceSpan` 等多表，降低早期复杂度。

### 2. LLM 可选 + 规则降级（核心设计）

- 后端解析层（简历/JD/优化建议）优先调用 LLM；未配置 `LLM_PROVIDER` 或 Key、或调用失败时，自动降级为基于技能本体的**规则解析**。
- 价值：保证无密钥也能离线跑通并通过测试，结果稳定可复现；同时为接入 OpenAI/Claude 预留统一入口（`services/llm.py`）。
- 缺口分析与路线生成采用**确定性规则编排**，不依赖 LLM，保证结果可解释、可验收（呼应 PRD §10.3）。

### 3. 单次分析采用一次性 `POST /api/analysis/run`

- 前端主流程一键提交 `resume_text + jd_texts`，后端在一次请求内完成解析→缺口→优化→路线并持久化，减少往返。
- 同时保留 PRD §12.1 中的 `resumes/parse`、`jobs/import` 等粒度接口，`analysis/run` 亦支持引用已存 `resume_id + job_ids`。

### 4. 技能本体 v0.1

- 以「别名表 + 树状分类 + 每技能学习模板/自检标准」的纯数据形式实现（`app/data/skills.py`），覆盖前端/后端/计算机基础/算法/工程化/软技能。
- 归一化区分中英文匹配：英文别名用非字母数字边界，避免 `ts`、`go` 等误匹配。
- 后续可平滑升级为数据库表或图谱（PRD §8）。

### 5. 可解释性落地（PRD §6.2）

每条结论都带来源：
- 缺口技能标注「来自哪几份 JD」（`required_by`）、出现频次、优先级、缺失/薄弱判定及原因；
- 已具备技能标注简历中的证据词；
- 「薄弱」判定标准：技能仅出现在技能罗列、缺少项目经历支撑。

## 工程说明

- 本地 Node 运行时放在 `.tooling/`（gitignored），由启动脚本自动优先使用，避免污染系统环境。
- 前端为单一 tsconfig + `vue-tsc --noEmit` 类型检查，未使用 project references，降低构建配置复杂度。

## 已知局限与后续 TODO

- 规则解析对格式混乱的简历/JD 抽取能力有限，建议配置 LLM 提升准确度。
- 技能本体覆盖面有限，需按首发岗位方向持续扩充别名与学习模板。
- PDF 仅支持文本型，扫描件需 OCR（暂不支持）。
- 尚未做用户体系与鉴权；历史记录为全局共享。
- 评测：PRD §13 建议的小样本人工评测尚未开展。
