# 前端性能审查 · 2026-06-12

> **审查方式**：多智能体三阶段——5 个维度并行静态分析（流式渲染路径 / 响应式与侦听开销 /
> 网络与持久化 / 包体与加载 / 内存与泄漏）→ 无头浏览器在**隔离栈**上实测关键假设（临时后端
> 7999 + 临时 sqlite + 临时前端 8471，不触碰开发数据库）→ 中高影响发现逐条对抗核实。
> 共 18 个智能体；全部关键结论附实测数字。
>
> **审查基线**：ChatView 拆分重构完成后（commit 57ba256）。Vue 3.5 + Vite 6，SSE 流式对话应用。
> **标准校准**：只报「真实用户可感知」的问题——长对话、长流式输出、大报告、频繁交互下的
> 卡顿/内存增长/流量浪费；不报微优化洁癖。
> **实测说明**：数字基于 dev 构建（无压缩），生产构建绝对值略好，但 O(n²)/线性增长的形态
> 与倍数关系不变。实测脚本（t1-t6）当时留存于 /tmp/op-focus-debug/，可按附录方法复建。

## 一、总体结论

**存在一个实测坐实的流式渲染 O(n²) 主瓶颈 + 一个一行可修的泄漏放大器，两者相乘构成
「流式输出越长越卡、应用越用越卡」的真实风险**；其余为中等的网络/持久化浪费（功能正确，
只是费流量、费 token、费服务端 CPU）。与此同时，渲染隔离架构、包体、SSE 解析、v-for key
策略等基本面非常健康——问题高度集中，修复性价比极高（前两项合计改动极小）。

| 维度 | 结论 |
|---|---|
| 流式渲染路径 | 架构健康，瓶颈集中在最后一环「文本→DOM」（O(n²) + 钩子放大） |
| 响应式与侦听 | 渲染隔离做得好，deep watch/持久化 ref 开销可忽略；问题同上归于 markdown 策略 |
| 网络与持久化 | 每轮全量 POST、路由往返重拉、卸载不中止 SSE——三处真实浪费 |
| 包体与加载 | 非常健康：首屏 92KB gzip（放宽标准的 18%），无 high/medium 发现 |
| 内存与泄漏 | DOMPurify 钩子无界累积（真泄漏）；计时器/监听器/watch 清理基本无问题 |

## 二、高危（流式渲染，两个问题相乘）

### P1. MarkdownText 每个 delta 全文重渲——O(n²) 主瓶颈 ✅核实 ✅实测

- **位置**：`MarkdownText.vue:28-32`（computed 全文 `marked.parse` + `DOMPurify.sanitize`）、
  `:37`（v-html 整块重建 DOM，无 diff）；上游 `useChatStream.ts:89-96` 每个 delta 直写
  响应式 text、`:85` 流式期 reasoning 默认展开（开销翻倍）。
- **实测**：500 个 SSE 事件（100 reasoning + 400 delta，10ms/事件）→ marked.parse 恰好
  500 次、sanitize 恰好 500 次；**累计解析入参 1,236,833 字符 vs 最终全文仅 ~10.8K 字符
  （平方级放大 ~115×）**；CDP LayoutCount 512（≈每 delta 一次整块重建+重排）。
  66KB 长输出（1000 delta）：单次 parse 四分位均值 0.71→1.62→2.46→3.51ms 线性爬升，
  事件吞吐被压到约 1/4（200 事件/秒 → ~48/秒）。
- **对照组**：同文本按 100ms 合帧（10 合 1）→ 解析 50 次、累计入参 127,473 字符、
  layout 63 次、脚本耗时 33ms→13ms、heap 增量 4MB→0。
- **可感知边界**：10KB 量级普通回复不掉帧（纯可量化浪费）；长输出（深度思考/长报告）真实可感。
- **建议**：流式 delta 按 ~100ms 合帧节流（在 useChatStream 合并写入，或 MarkdownText
  对 streaming 态节流）；中期可做按块增量渲染。

### P2. DOMPurify 全局钩子无界累积——泄漏 + 把 P1 越用越放大 ✅核实 ✅实测

- **位置**：`MarkdownText.vue:17/20-25`——`marked.setOptions` 与 `DOMPurify.addHook` 写在
  `<script setup>` 体内，**每挂载一个实例就向 DOMPurify 全局钩子数组 push 一个永不移除的
  钩子**（dompurify 3.x 无去重，每实例闭包不同；全 src 无 removeHook）。
- **实测**：钩子数严格等于历史创建过的实例数——启动 0 → 载 5 轮会话 10 → 载 50 轮会话 100
  → 重复载入 3 次 300；切到 /usage（页面 0 个实例）仍是 300，**只有整页刷新才清零**（SPA
  设计下用户通常不刷新）。同一段 120 元素 HTML 的单次 sanitize：0.638ms（0 钩子）→
  11.70ms（300 钩子，**18.3×**），且直接叠乘在 P1 的每-delta 热路径上。
- **建议**：`setOptions`/`addHook` 移到模块顶层（一行级修复）。核实者注：单独看可降为
  medium，但与 P1 相乘后构成「越用越卡」，保持高优先级。

## 三、中等（网络与持久化的浪费）

### P3. 每轮对话结束全量 POST 整个会话——累计 O(轮数²) ✅核实 ✅实测

- `useConversationPersistence.ts:78-84` 每轮 persist 无 diff/防抖，全量 `serializeTurns` +
  完整 context（简历全文 + 全部 JD）POST；report 块自带完整 result JSON，与 analysis_runs
  表双份入库后仍随每轮重传。
- **实测**：载入 420KB 会话后连发 3 条 5 字短消息 → 3 次 POST 实测 433,674 / 433,860 /
  434,046 字节，单次有效增量仅 ~190 字节——**上送有效率 0.04%**，3 条短消息累计上传 1.3MB。
- 建议：增量/防抖保存；报告 result 只存 analysis_run_id 引用（回放时按需取）。属架构性改动，可后置。

### P4. 路由切走不中止在途 SSE 流——后台白烧 token ✅核实 ✅实测

- `useChatStream.ts:45/200-202` abortController 私有且仅「停止」按钮触发；`ChatView.vue`
  onUnmounted 不中止；无 onBeforeRouteLeave/KeepAlive。
- **实测**：发起 30 秒流式（601 事件），第 51 个事件时切到 /usage——流持续接收 30.1 秒
  收完 601/601，结束后在 /usage 页发出孤儿 POST 落库。真实场景＝切走后 LLM 继续生成
  1-3 分钟、token 照常计费。
- 缓解（核实者纠正）：后台跑完的输出会经孤儿 persist 落库 + SideNav 刷新，可从侧栏点开找回，
  不算丢失——浪费的是 token 与带宽，不是数据。
- 建议：onScopeDispose/onUnmounted 时 abort（注意保留「停止按钮」语义：手动停止落一次保存
  的红线行为不变；卸载场景可选择同样落一次保存后中止）。

### P5. 无 keep-alive 的路由往返全量重拉 ✅核实 ✅实测

- 「查看完整报告 → 返回对话」：ResultView onMounted 无条件 `getAnalysis` 回源（报告 result
  此刻已完整在内存的 report 块里）；返回时 ChatView 全新实例使去重守卫失效，整个会话重拉。
- **实测**：往返 ×5 = GET /api/analysis/1 ×5 + GET /api/conversations/1 ×5（每次 708,654
  字节）——同一份内容冗余下载 3.5MB，每次返回伴随 348ms 全量重渲染。
- 建议：ChatView 加 KeepAlive 或会话级内存缓存；ResultView 支持从路由 state 接收已有 result。

### P6. 回放/续聊载入全量同步渲染，折叠思考块 v-show 驻留 ✅核实 ✅实测

- `AssistantBlocks.vue:98` 折叠 reasoning 正文用 v-show 包 MarkdownText——挂载即解析；
  还原时全折叠（`chatModel.ts:237`），解析成本全为无效付出。
- **实测**：50 轮 420KB 会话载入 → 100 次 parse（**50 次来自从未展开的折叠思考块**）、
  DOM 29,790 节点、单次 longtask 284ms、导航→首帧 348ms、heap 11.2MB。
  v-show→v-if 对照：parse 100→54 次、节点 -59%（12,092）、heap -3.5MB，**但首帧仅 -10%**
  （瓶颈是一次性同步渲染全部轮次本身）。
- 建议：reasoning 正文改 v-if（立省内存与解析）；首帧卡顿要根治需分帧/懒渲染历史轮次，可后置。

### P7. SideNav 每轮重拉会话列表（核实者建议降为 low）✅核实 ✅实测

- 每轮 persist 后无条件 `notifyConversationsChanged` → SideNav 重拉 GET /api/conversations。
- **实测**：前端流量小（37 会话 5.3KB）；后端为算 turn_count 反序列化**全库**会话 JSON，
  37 会话 29ms → 57 会话（库 29.5MB）83ms，随库容量线性。个人项目量级下属轻微浪费。
- 建议：后端列表接口不反序列化 turns（存冗余 turn_count 列或 json_array_length）。

## 四、被对抗核实驳回的猜想（留档防重提）

**「百轮长会话下流式期每帧强制 layout 随 DOM 总量变慢」→ 驳回。**
实测 29,790 节点 DOM + 100 个钩子下，同一 500 事件流式总耗时与 1,395 节点小会话**完全一致**
（5,020ms）、0 longtask、0 掉帧。双重天然化解：① Vue 组件级精确依赖——每 delta 只重渲当前
气泡，历史轮 props 引用不变直接跳过；② 浏览器增量布局。但附带主张成立：DOM 规模实测
~300 节点/助手轮（百轮外推 ~6 万）、键入 20 字符延迟 58ms（5 轮）→157ms（50 轮，2.7×），
临界可感。

## 六、检查过且确认健康的方面（防止重复怀疑）

- **SSE 分帧解析**：indexOf/slice 即时消费、TextDecoder 流式解码，缓冲不积累，无 O(n²)。
- **渲染隔离**：每 delta 只重渲当前流式 turn 的 AssistantBlocks；ChatView 编排层与
  ChatComposer（1346 行）不随 delta 重渲；latestReport 只追踪 blocks 结构不追踪 text。
- **v-for key 策略**：turns/blocks/steps 的 index key 在「尾部追加+尾部截断」写入模式下安全。
- **滚动指纹**：只拼条数与末块长度数字，不拷贝全文。
- **包体非常健康**：首屏（index 45.2 + ChatView 14.1 + AssistantBlocks 24.2 + ScoreRing 0.9）
  ≈ **92KB gzip**（JS 83.4 + CSS 8.1），仅放宽标准 500KB 的 18%；7 个路由全部真实懒加载；
  marked+dompurify（~19KB gz）未进主包、归属合理；AnalysisReport 不在首屏依赖链；
  无未用依赖；vite 默认分包+minify 在此规模下足够。
- **无轮询**：progress/会话列表均为信号驱动惰性刷新。
- **手动停止的资源释放**：abort 后 streamChat 安静返回、controller 置 null，无泄漏。
- **deep watch 开销可忽略**：llmConfig（6 个短字段）、tone/折叠态等 usePersistedRef 写回
  均微秒级；UsageView filters watch 空载。
- **事件监听/watch 清理**：全 src 仅 1 处模块级 matchMedia 监听（刻意单例）；模板绑定随卸载
  自动解绑；composable watch 均在 setup 内创建随 scope 停止；useCountUp 正确 cancelAnimationFrame。
- **AnalysisReport**：buildMarkdown 仅复制/下载时调用；报告卡默认折叠 + v-if，不在渲染热路径。

## 七、修复性价比排序

1. **一行**：MarkdownText 的 `setOptions`/`addHook` 移到模块顶层（P2）→ 消除 18× sanitize
   放大与无界泄漏。
2. **小**：流式 delta ~100ms 合帧节流（P1）→ 消除 O(n²) 主瓶颈（与 1 合计消除流式全部高危）。
3. **小**：折叠 reasoning 正文 v-show→v-if（P6）→ 长会话省 59% DOM 节点、3.5MB 内存。
4. **小**：ChatView 卸载时 abort 在途流（P4）→ 不再白烧 token。
5. **架构性，可后置**：persist 增量/防抖 + 报告去重入库（P3）、路由往返会话缓存/KeepAlive（P5）、
   会话列表接口免反序列化（P7）、历史轮分帧渲染（P6 根治）。
