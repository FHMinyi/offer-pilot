# 上下文系统设计与缓存评审 · 2026-06-12

> **评审方式**：4 个并行读码智能体分头建立事实清单（后端对话链路 / 状态持久化 / 前端上下文构造 / 设计文档意图，全部带 file:line 引用）；缓存相关结论由主会话直接读码核证（`agent.py` 的 `_system_prompt` 与回合循环、`llm.py` 全文 grep `cache_control`、`usage.py` 的 hit/miss 归一化）。
>
> **评审范围**：chat 主链全链路（前端组装 → 请求体 → system prompt → 回合循环 → 工具回灌 → 持久化与恢复），以及长期状态地基的读写边界。不含 mastery/interview 等旁路 LLM 调用的内部质量。
>
> **校准**：按项目真实定位（个人演示型开源项目，见 `docs/行动计划.md`）——缓存问题之所以仍列高位，是因为它同时影响演示成本与"工程故事"的说服力，且修复刀口极小。

---

## 一、总体结论

**这是一套"后端零会话状态、前端每轮全量重传"的显式上下文设计**：对话上下文的真值在前端内存里，后端每轮现拼 system prompt，数据库只扮演两个角色——会话快照仓库（Conversation 整存整取）+ 一个跨轮分析锚点（`analysis_run_id`）。结构清晰、边界自觉（欠账均在文档显式记账），但存在 **三个相互独立的结构性缓存失效问题**，叠加效果是：跨轮 prompt cache 命中率基本只剩 system prompt 开头几百 token 的固定指令，简历、JD、全部对话历史每轮按 miss 原价重付。配合"历史无裁剪 + 全量重传"的 O(n²) 输入增长，长对话的 input 成本中约九成本可由缓存吸收的部分目前全额支付。

另一条主线缺口与缓存无关但同根同源：**长期状态地基"写侧已通、读侧未接"**——journey/Task/CheckIn 写闭环完整，但没有任何路由把这些状态注入 chat prompt，AI 不知道用户的过去（即差距分析所称"灵魂≈0"的技术落点）。

## 二、设计现状（六层拆解）

### 1. 请求级：前端是上下文的唯一持有者

每轮 `POST /api/chat/stream` 五字段（`backend/app/schemas.py:151-159`）：

- `messages` —— 全量历史投影。前端 `toPayloadMessages` 把 assistant 的思考/工具/报告块丢弃、文本块拼成一条（`frontend/src/shared/chatModel.ts:106-112`），**无任何条数/长度裁剪**。
- `context` —— `snapshotChatContext` 快照（`chatModel.ts:119-138`）：`resume_text` / `jd_texts[]` / `target_role` / `weeks` / `analysis_run_id` / `tone` / `persona`（预留，前端从不写入）。
- `reasoning_effort` / `client_time`（前端 `new Date().toLocaleString('zh-CN',{hour12:false})`，秒级，`useChatStream.ts:75`）/ `llm_override`（BYO 配置，api_key 仅存浏览器）。

后端不维护 session：`chat.py:32-33` 每次从请求体 dump，用完即弃。`user_id` 由 `X-Device-Id` 解出后作显式参数穿透全链（`chat.py:54-56`），流式生成器用 `contextvars.copy_context()` 专属 ctx 驱动，解决 Starlette 线程池逐步迭代下 override/usage 归属的线程漂移（`chat.py:45-62`）。

### 2. Prompt 组装层：每轮现拼，截断只发生在这里

`_system_prompt`（`agent.py:171-221`）段序为：**人设第一句 → "两步走"流程指令（静态）→ 风格行 → tone 指令 → 当前时间 → 目标岗位/周数 → 材料段**（简历截前 4000 字符、每条 JD 截 2000，缺材料替换为索取提示，`agent.py:185-194`）。

值得记录的正确设计：截断只影响"给模型看"的 prompt 注入；`analyze_match` 工具实际取材用 context/参数**全文**（`agent.py:406-413`），分析质量不受截断影响——prompt 里那份只为让模型知道"材料在手，别再索取"。

### 3. 回合内状态：只活在一次 SSE 流里

`_MAX_STEPS=5` 工具循环上限；`did_analyze` 硬闸门（同轮 analyze_match 后拦截 generate_plan，强制反问用户，`agent.py:288-303`）；`spoke` 兜底固定反问文案；工具结果以 tool 消息回灌（分析结果只回 `_match_summary` 摘要：匹配度 + 前 4 缺口，非完整报告）；轮内 token 累加器。轮内 `llm_messages` 纯追加（`agent.py:272-286`）。

### 4. 跨轮锚点：唯一的 DB 回读

`analysis_run_id`：analyze_match 落库 → report 事件下发 → 前端记住并随后续每轮 context 回传 → generate_plan 用它 `db.get(AnalysisRun)` 读回上次分析（`agent.py:455-462`）。除此之外 chat 主链不从 DB 读任何内容进 prompt。

### 5. 持久层：DB 是前端内存的快照仓库

每轮流结束自动 upsert Conversation（turns + context 快照，`useConversationPersistence.ts:72-93`）；恢复完全前端驱动（`/?c=<id>` 加载，`ChatView.vue:298-308`）。localStorage 只存偏好（`op.tone` / `op.llm-config` / `op.device.id` 等）。

### 6. 长期状态地基：写通读未接

JourneyState/Task/CheckIn 写闭环完整（物化、replan、面经回灌、掌握度），但无任何代码把这些状态汇总注入 chat prompt；唯一接近"记忆→LLM"的是 mastery 判定 prompt 里的单任务 `Task.title/skill_key`（`mastery.py:152-157`）。地基方案当初预留了读取点（`docs/方案_里程碑一_地基层_2026-06-05.md:378`），属显式记账的待接线。

## 三、问题清单

### 高（缓存三连，相互独立、效果叠加）

**P1. 秒级时间戳插在 system prompt 中部，每轮必断缓存**

- `client_time` 每轮由前端现取、精确到秒（`useChatStream.ts:75`），注入位置在 `agent.py:218`——它前面只有固定指令 + tone（约几百 token），**后面是目标岗位、简历 4000 字符、N×2000 字符 JD、再加全部对话历史**。
- 前缀缓存按最长公共前缀匹配，第一处差异之后全部作废 → 每轮缓存在时间戳处折断，体量最大的材料段与历史**永远 miss**。
- 这是三个问题里杀伤最大、修复最便宜的一个。

**P2. 历史是有损重投影，不是 append-only 转录**

- 第 N 轮模型实际见到 `[..., assistant(tool_calls), tool 结果, assistant 文本]`；第 N+1 轮发出的却是 `[..., assistant 合并文本]`——前端丢弃工具/思考块（`chatModel.ts:106-112`），后端只放行 user/assistant（`agent.py:240-245`）。
- 序列在**第一个用过工具的轮次**分叉。即使 P1 修复，缓存也会在那里断（只能命中到分叉点之前）。

**P3. Anthropic 协议路径根本没启用缓存**

- `llm.py` 全文无 `cache_control`；Anthropic prompt caching 必须显式打断点，不打即 0 命中。`usage.py:101-106` 已备好 `cache_read_input_tokens` 归一化，但该路径下永远是 0。
- OpenAI/DeepSeek 兼容路径是 ≥1024 token 前缀自动缓存，所以目前还能蹭到一点。

### 中

**P4. 历史无裁剪 + 全量重传 → 输入 token O(n²) 增长**

- 累计 input ≈ 每轮(系统提示+材料) + Σ历史。粗算 20 轮、平均每轮新增 500 token、材料段 3k token：总 input ≈ 165k token 全按 miss 价；缓存正常工作时其中大头可按约 0.1× 读价支付（粗算口径，量级正确即可）。
- 与《前端性能审查_2026-06-12》的"每轮全量 POST（上送有效率 0.04%）"是同一现象的两面：那边算的是网络字节，这边算的是 token 账单。

**P5. 裸 `/` 刷新丢上下文（数据已存、不自动恢复）**

- 自动保存成功后不把路由改写为 `/?c=<id>`（`useConversationPersistence.ts:87-89`），裸 `/` 上刷新 → turns/context/currentRunId 内存态全丢，界面变空白新对话；需手动从侧栏/历史"继续"。

**P6. 状态地基读侧未接（"灵魂"缺口的技术落点）**

- 见 §二.6。AI 不知道用户打卡几天、完成了哪些任务、面试暴露过什么盲区——"数月级陪伴管家"愿景在对话层无感知。

### 低

- **P7.** `JourneyState.persona/tone` 列（`models.py:144-145`）写而不读——chat 实际用前端传的 tone/persona，DB 列成摆设；与差距分析曾批评的"structured 写而不读"同款气味。
- **P8.** tone 指令位于材料段之前（`agent.py:217`），滑块一动也会切断材料段之后的缓存（频率低，随 P1 的段序重排顺手解决）。

## 四、修改建议（按性价比排序）

| # | 改动 | 刀口 | 收益 |
|---|---|---|---|
| R1 | 时间戳降粒度到"日期+小时"，并与 tone 一起挪到 system prompt **末尾**（或移入最后一条 user 消息） | `_system_prompt` 几行 | 单独消灭 P1/P8，材料段进入可缓存区 |
| R2 | 段序重排为"静态指令 → 材料（稳定）→ 易变项（tone/时间）"，从稳到变排列 | 与 R1 同一处 | 缓存前缀最大化，属 R1 的完整形态 |
| R3 | anthropic 路径加 `cache_control` 断点（system 末尾 + 最后一条消息各一个） | `llm.py` 转换层十几行 | 消灭 P3，anthropic 路径从 0 命中变正常 |
| R4 | 裸 `/` 首次保存成功后 `router.replace({query:{c:id}})` | 前端一行级 | 消灭 P5，刷新可恢复 |
| R5 | 读侧注入：`_system_prompt` 增加一段从 journey/progress 聚合的"旅程现状"（当前周/完成率/streak/最近盲区，几十 token） | 后端一个聚合函数 + prompt 一段 | 点亮 P6，是"有记忆的陪伴"最薄的一刀，也是简历故事的关键差异点 |
| R6 | 历史保真转录：回传并重建工具轮次（assistant tool_calls + tool 消息），或后端持久化 LLM 原始消息序列 | **大**（前后端协议与存储都动） | 彻底消灭 P2；建议最后做，先吃 R1-R3 的九成收益 |
| R7 | 历史裁剪/摘要（超过阈值后旧轮次折叠为摘要） | 中 | 缓解 P4 的 O(n²)；注意与缓存互斥面——裁剪点变动也会断缓存，宜按"整段冻结"方式裁 |

**建议的最小落地组合：R1+R2+R3（合计百行内），跨轮命中率从"只剩固定指令头"变为"除最新轮外全命中（到第一个工具轮为止）"。** R5 与缓存无关但同样刀口小、叙事价值高，适合并入同一个里程碑。

## 五、检查过且健康的方面（防止未来重复怀疑）

- 轮内多步工具循环缓存友好：`client_time` 每轮取一次、轮内不变，`llm_messages` 纯追加——usage 面板里现有的 `input_hit` 基本全来自这里（且仅 OpenAI 系路径）。
- 截断与取材分离（§二.2）：4000/2000 截断不影响分析质量。
- token 统计口径正确：`input_hit/input_miss` 全链路统一命名，OpenAI/DeepSeek/Anthropic 三家语义差异在 `usage.py:72-106` 归一化处理无误——观测体系已就绪，修完 R1-R3 当天就能在面板看到 hit 占比变化。
- `did_analyze` 硬闸门 + `spoke` 兜底：不依赖模型自觉守"两步走"流程，结构性保证。
- `user_id` 显式穿透与 contextvars 专属 ctx：流式线程漂移下归属不串。
- 降级路径 `_scripted_turn`：无 LLM 时规则闭环可跑，零配置演示承诺成立。
